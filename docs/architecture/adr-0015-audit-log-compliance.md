## Context

### Problem Statement
Several parent operations are sensitive enough to warrant an immutable audit trail: login, child access token regeneration, check-in revocation, reward approval/rejection, and (Phase 2) data export. The system needs a single table that:

1. Records **who** (actor_role, actor_id), **did what** (action), **to what** (target_type, target_id), **when** (createdAt), with optional context (`metadata`, `ip_address`).
2. Is **append-only** — no UPDATE or DELETE path; rows can only be inserted.
3. Is scoped by `family_id` for multi-tenant isolation (ADR-0006) but supports system-level events (no family).
4. Uses `BIGSERIAL` PK (per ADR-0005 Conflict #5 resolution) for write performance and small index footprint.

### Constraints
- Audit log is the **only** table that deviates from the project's UUID PK convention (ADR-0005 §Conflict #5 documents the rationale: append-only, write-throughput matters more than GUID randomness).
- Audit log must NOT be exposed via a public REST endpoint — it is operator-only.
- Audit log writes must never block the main transaction — a failed audit write should not roll back a successful reward approval.
- IP address is captured from `request.ip` (Fastify) — `X-Forwarded-For` is trusted only behind Railway's proxy (ADR-0008).

### Requirements
- **TR-audit-001**: Audit log records parent-critical operations: login, token regeneration, check-in revocation, reward approval/rejection/fulfillment, child creation, task creation/update/delete, reward creation/update/delete.
- **TR-audit-002** (implied): Audit log is append-only; no UPDATE or DELETE SQL is generated by the application.
- **TR-audit-003** (implied): Audit log entries are scoped by `family_id` for multi-tenant isolation.
- **TR-audit-004** (implied): Audit log is queryable by operators (Phase 2 admin tool), not by parents or children via public API.

## Decision

### 1. Schema (already in `schema.ts`)

```typescript
export const auditLogs = appSchema.table('audit_logs', {
  id: bigserial('id', { mode: 'bigint' }).primaryKey(),
  familyId: uuid('family_id').references(() => families.id, { onDelete: 'cascade' }),
  actorRole: text('actor_role').notNull(),  // 'parent' | 'child' | 'system'
  actorId: uuid('actor_id'),
  action: text('action').notNull(),          // e.g., 'parent.login', 'reward.approve'
  targetType: text('target_type'),           // e.g., 'reward_redemption'
  targetId: uuid('target_id'),
  metadata: jsonb('metadata'),
  ipAddress: text('ip_address'),
  createdAt: timestamp('created_at', { withTimezone: true }).notNull().defaultNow(),
}, (t) => ({
  actorRoleCheck: check('check_audit_actor', sql`${t.actorRole} IN ('parent', 'child', 'system')`),
  familyIdx: index('idx_audit_family').on(t.familyId, t.createdAt.desc()),
}));
```

### 2. Action Naming Convention

Actions follow a `<entity>.<verb>` lowercase string convention:

| Action | Triggered By | Target Type |
|--------|-------------|-------------|
| `parent.login` | Successful parent login | `parent` |
| `parent.register` | New family registration | `family` |
| `child.create` | Parent creates a child profile | `child` |
| `child.token_regen` | Parent regenerates child access token | `child` |
| `task.create` / `task.update` / `task.delete` | Parent task CRUD | `task` |
| `reward.create` / `reward.update` / `reward.delete` | Parent reward CRUD | `reward` |
| `checkin.revoke` | Parent revokes a check-in | `checkin` |
| `redemption.approve` / `redemption.reject` / `redemption.fulfill` | Parent updates redemption status | `reward_redemption` |
| `data.export` (Phase 2) | Parent exports family data | `family` |

### 3. Logging Helper

```typescript
// apps/api/src/server/services/audit.ts
export async function writeAudit(
  familyId: string | null,
  actorRole: 'parent' | 'child' | 'system',
  actorId: string | null,
  action: string,
  options: {
    targetType?: string;
    targetId?: string;
    metadata?: Record<string, unknown>;
    ipAddress?: string;
  } = {}
): Promise<void> {
  try {
    await db.insert(schema.auditLogs).values({
      familyId,
      actorRole,
      actorId,
      action,
      targetType: options.targetType || null,
      targetId: options.targetId || null,
      metadata: options.metadata || null,
      ipAddress: options.ipAddress || null,
    });
  } catch (err) {
    // Log failure but do not throw — audit must not block the main transaction
    logger.error({ err, action }, 'audit_log_write_failed');
  }
}
```

### 4. Non-Blocking Write Strategy

Audit log writes are **fire-and-forget** — the helper awaits the insert but catches and logs any error. This trades strict consistency (an audit row might be missing if the insert fails) for availability (the main operation succeeds regardless). The trade-off is acceptable because:

- The audit log is a forensic tool, not a transactional one.
- The main operation (e.g., reward approval) has already committed; a missing audit row degrades forensic granularity but does not corrupt business state.
- For strict audit requirements (e.g., financial compliance), Phase 2 may upgrade to a transactional outbox pattern.

### 5. IP Address Capture

- `request.ip` is read from the Fastify request object.
- Behind Railway's proxy, `request.ip` resolves to the client IP via `X-Forwarded-For` (trusted proxy config in `app.ts`).
- Direct browser access (dev) returns `127.0.0.1` — acceptable for local development.
- IP addresses are **not** encrypted — they are operational metadata, not PII per PRD §7.3.

### 6. Retention

- MVP: No automatic retention policy. Audit logs accumulate indefinitely.
- Phase 2: A scheduled job (per ADR-0010) will archive entries older than 365 days to cold storage (S3) and delete them from the primary table.

## Consequences

### Positive
- ✅ Append-only design guarantees tamper-evidence from the application layer (DB-level triggers can enforce this in Phase 2).
- ✅ `BIGSERIAL` PK optimizes for write throughput and small index footprint (per ADR-0005 Conflict #5 rationale).
- ✅ `jsonb` metadata column allows flexible context capture without schema migrations.
- ✅ Non-blocking write strategy ensures audit never breaks business operations.

### Negative
- ⚠️ Non-blocking writes mean audit log is eventually consistent with the main transaction — a missing audit row is possible under DB pressure.
- ⚠️ No public REST endpoint means operators must query the DB directly in MVP; Phase 2 adds an admin UI.
- ⚠️ No automatic retention policy in MVP — audit log can grow unbounded. Phase 2 adds archival.

### Neutral
- `actor_id` is nullable to support system-initiated actions (e.g., scheduled job cleanup) where no human actor exists.

## Risks

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| Audit log write fails silently, masking security incidents | MEDIUM | Failed writes are logged via `pino` at `error` level; Sentry alerting (Phase 2) can trigger on `audit_log_write_failed` |
| DB-level UPDATE/DELETE bypasses append-only contract | LOW (no DB access in MVP) | Phase 2 will add a PG trigger `BLOCK UPDATE/DELETE ON audit_logs` |
| Audit log grows unbounded in MVP | MEDIUM | Phase 2 retention job + archival to S3 |
| IP address captured behind untrusted proxy | LOW | Railway proxy is trusted; `trustProxy` config in `app.ts` is set explicitly |

## Open Questions

None — all decisions are pinned for MVP Phase 1. Phase 2 will add DB-level tamper-evidence triggers and an admin query UI.
